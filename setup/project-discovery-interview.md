# Project Discovery Interview Rules

## Objective

Guide a high-signal interview that starts from free prose and converges quickly to implementation-ready context.

## Behavior rules

1. Intake first, questionnaire later.
2. Summarize before asking follow-up questions.
3. Ask only gap-closing questions.
4. Prefer assumptions marked as assumptions over endless questioning.
5. Keep tone collaborative and concise.

## Mandatory interview sequence

### Step 1 - Intake libre

Ask for all available context in natural language:

- product vision
- business model
- operational flows
- known constraints
- must-have scope and non-goals

Do not request strict formatting.

### Step 2 - Structured understanding

Return a compact synthesis with these sections:

- understood facts
- detected entities and vocabulary
- detected surfaces and users
- inferred auth/tenancy/capabilities
- ambiguous points

### Step 3 - Minimal questions

Ask only unresolved critical questions.

Guidelines:

- prioritize by impact (identity/domain/surfaces/auth first)
- ask in small batches (3-7 max per round)
- include why each question matters
- include a recommended default when reasonable

### Step 4 - Closure and assumptions

When some details remain open but non-blocking:

- declare explicit assumptions
- mark them as provisional
- continue synthesis if completion gate still passes

## Anti-patterns (forbidden)

- starting with a full category dump of questions
- asking questions already answered in user prose
- pretending certainty where there is ambiguity
- generating docs before confirming critical unknowns
- over-interviewing low-impact details

## Response skeleton (recommended)

Use this shape after analyzing user prose:

1. "What I already understood"
2. "What seems to be the product model"
3. "What is still missing"
4. "Minimal questions to close gaps"

Keep it short and decision-oriented.
